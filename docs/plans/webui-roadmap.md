# DocPage2MD WebUI Roadmap

## 目标

当前 Tkinter GUI 继续作为轻量桌面入口维护。中期新增本地 WebUI，用更清晰的工作台承载批量文件、模型管理、实时进度、Markdown 预览和历史任务。

WebUI 不替代 CLI。CLI 仍是稳定自动化入口，WebUI 只负责任务配置、状态订阅和结果浏览。

## 推荐架构

- 后端：FastAPI
- 实时状态：SSE 或 WebSocket
- 前端：React、Svelte 或 Vue，优先选择团队维护成本最低的一种
- 任务执行：后端任务队列包装现有 `docpage2md.py` / `docpage2md_app` 能力
- 桌面封装：后期需要原生桌面体验时再用 Tauri 包装本地 WebUI

关键原则：

- 后端跑任务，前端只订阅状态，避免 UI 卡顿。
- 进度按真实文件、页数、裁剪块和并发状态计算，不靠固定动画。
- 日志使用虚拟滚动，避免大日志拖慢页面。
- 模型配置只保存环境变量名，不保存 API Key 明文。
- WebUI 不读取、不删除 `markdown_output/已归档`。

## 页面划分

### 输入页

- 拖拽 PDF/Office/图片文件。
- 支持单文件、多文件、文件夹、URL、MinerU artifact。
- 显示页码范围、输出目录、任务名。
- 文档类型只作为推荐预设，用户可覆盖处理模式和模型档位。

### 模型页

- 当前生效 Vision / Brain 模型。
- 官方模型列表，显示 provider、模型 ID、价格、Key 状态。
- 第三方模型管理，支持新增、编辑、删除、验证、批量导入。
- API Key 环境变量检查和写入 Windows 用户环境变量。
- 保留 `cheap / balanced / accurate` 默认档位，同时允许完全自定义。
- 价格表支持手动刷新和后续实时刷新：DeepSeek 可抓官方文档，Qwen/百炼优先读取本地精选表并允许用户更新。

### 运行页

- 总进度、每个 PDF 进度、每页状态。
- Vision 并发、Brain 并发、当前正在处理的页和裁剪块。
- ETA 基于真实完成页/块和运行耗时估算。
- 中文实时日志。
- 开始、停止、重试失败任务。

### 输出页

- `Slide_XX.md` 预览。
- `{doc_name}_FULL.md` 预览。
- 页面图、crop assets、`run_report.json`。
- 打开输出目录。
- 显示 validator 结果，但不把诊断文本混入最终 Markdown 正文。

### 历史页

- 历史任务列表。
- 状态、耗时、输出路径、模型配置、成本估算。
- 记录估算价格版本、实际 usage 和估算偏差，方便后续校准默认输出 token。
- 重跑入口。

## 分阶段实施

### Phase 1：保持 Tkinter 可用

- 中文化运行页和模型页。
- 恢复 GUI 成本估算。
- 放大日志窗口。
- 保留并行参数和当前 CLI 覆盖参数。
- 明确 WebUI 是后续计划。

### Phase 2：本地 WebUI 原型

- FastAPI 启动本地服务。
- 前端实现输入页、模型页、运行页。
- 后端复用现有 CLI 构造和任务执行逻辑。
- SSE/WebSocket 推送中文进度和日志。
- 模型价格表先提供本地刷新按钮，实时刷新作为可选后台任务，不阻塞主流程。

### Phase 3：结果浏览和历史任务

- Markdown 预览。
- `run_report.json` 可视化。
- 历史任务索引。
- 失败任务重跑。

### Phase 4：桌面封装

- 用 Tauri 包成本地应用。
- 自动启动后端服务。
- 系统托盘、打开输出目录、环境变量管理。

## 不采用节点式工作流

DocPage2MD 的主要用户路径是“选文件、选模型、开始、看结果”。不采用 ComfyUI 式节点画布，避免把简单批处理任务变成工作流编排工具。
